### 2.1 TDD 讲解深度策略

| 层级 | 内容 | 位置 |

**常见合理化反驳**:
1. **"这个功能太简单，不需要测试"**
   - 反驳: 简单功能最容易被忽略改动，测试防止回归

2. **"我先写代码，稍后补测试"**
   - 反驳: 99% 的情况"稍后"永远不会来，这是伪 TDD

3. **"写测试浪费时间"**
   - 反驳: TDD 减少调试时间，净加速开发

4. **"测试代码也是代码，也需要测试吗？"**
   - 反驳: 测试代码不需要 TDD，但需要验证有效性

### 2.2 双阶段审查流程

[技术领域专家要求] 必须展示两阶段审查的区别

**阶段 1: Spec Compliance（规范符合性）**
- 目标: 验证实现是否符合设计规范
- 检查: 功能完整性、接口一致性、需求覆盖
- 输出: PASS/FAIL + 偏差清单

**阶段 2: Code Quality（代码质量）**
- 目标: 评估代码可维护性和最佳实践
- 检查: 命名、结构、性能、安全性
- 输出: DONE/DONE_WITH_CONCERNS/CHANGES_REQUESTED

**可视化**:
```text
双阶段审查流程

┌─────────────────────────────────────────┐
│ 阶段 1: Spec Compliance                 │
│ - 检查: 实现是否符合规范                 │
│ - 标准: 规格文档、需求清单               │
└────────────┬────────────────────────────┘
             │
             │ ✅ PASS
             ▼
┌─────────────────────────────────────────┐
│ 阶段 2: Code Quality                   │
│ - 检查: 代码质量、最佳实践               │
│ - 标准: 团队编码规范、最佳实践           │
└────────────┬────────────────────────────┘
             │
             ├─ DONE (无问题)
             ├─ DONE_WITH_CONCERNS (小问题)
             └─ CHANGES_REQUESTED (大问题)
```

### 2.3 审查状态机

[系统架构师要求] 应展示审查循环的终止条件

**状态定义**:
- **DONE**: 审查通过，可以合并
- **DONE_WITH_CONCERNS**: 通过但有顾虑，需讨论
- **CHANGES_REQUESTED**: 需要修改后重新审查

**状态转换**:
```text
┌──────────────┐
│ 初始提交      │
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ Spec Review  │ ← 阶段 1
└──────┬───────┘
       │
       │ FAIL → 需修改实现
       │ PASS │
       ▼      └─────────────┐
┌──────────────┐              │
│ Code Review  │ ← 阶段 2     │
└──────┬───────┘              │
       │                      │
       ├─ DONE ──────────────→ 合并
       ├─ DONE_WITH_CONCERNS ─→ 讨论
       └─ CHANGES_REQUESTED ──┘
```

## 附录 A: TDD 合理化反驳示例

```markdown
## 常见 TDD 合理化及反驳

### 1. "这个功能太简单，不需要测试"

**反驳**:
- 简单功能最容易被忽略改动
- 测试防止无意中的回归
- 测试即文档，说明"简单"的行为

**示例**:
```typescript
// "简单"的加法函数
function add(a: number, b: number): number {
  return a + b;
}

// 测试防止这样的改动:
// function add(a: number, b: number): number {
//   return a - b; // 符号改错了
// }
```

### 2. "我先写代码，稍后补测试"

**反驳**:
- 99% 的情况"稍后"永远不会来
- 这是伪 TDD，不是真正的 TDD
- 先写代码就失去了"测试驱动"的价值

**数据**:
- 研究显示"稍后补测试"的完成率 < 10%
- 真正的 TDD：测试先于实现

### 3. "写测试浪费时间"

**反驳**:
- TDD 减少调试时间（净加速）
- 测试即文档，减少沟通成本
- 重构时有安全网

**对比**:
```
无 TDD:
- 写代码: 1 小时
- 调试: 2 小时
- 修复回归: 1 小时
- 总计: 4 小时

TDD:
- 写测试: 30 分钟
- 写代码: 30 分钟（测试驱动，更快）
- 调试: 0 分钟（测试通过即正确）
- 总计: 1 小时
```

### 4. "测试代码也需要测试吗？"

**反驳**:
- 测试代码不需要 TDD
- 但需要验证测试有效性
- 关键：测试必须先失败（证明能捕获错误）

**示例**:
```typescript
// ❌ 伪测试（从未失败）
test('add returns sum', () => {
  expect(add(2, 2)).toBe(4); // 从未失败过
});

// ✅ 真实测试（先失败，后修复）
test('add returns sum', () => {
  // 第一次运行失败（add 未实现）
  expect(add(2, 2)).toBe(4);

  // 实现 add
  // 第二次运行通过
});
```
```
