## 1. 需求摘要

本功能通过完整的端到端示例，展示 SuperPower 在实际开发场景中的应用，包括 brainstorm（构思）、planning（规划）、development（开发）三大核心工作流。

**MUST**:
- 覆盖 TDD、Debug、Feature 三大核心工作流
- 使用真实项目场景
- 展示完整流程（问题 → 方案 → 验证）
- 每个工作流可独立运行

**SHOULD**:
- 提供渐进式真实度（初级→高级）
- 展示多 Skill 协作
- 包含常见错误处理

**MAY**:
- 添加挑战练习
- 提供扩展工作流

**MUST NOT**:
- 使用玩具示例（过度简化）
- 跳过验证步骤
- 假设用户有深入理解

### 2.4 TDD 工作流深度设计

[技术领域专家要求] 必须展示 RED-GREEN-REFACTOR 完整流程

**RED 阶段**:
- 先写失败的测试
- 验证测试失败
- 理解失败原因

**GREEN 阶段**:
- 编写最小代码使测试通过
- 不追求完美，追求通过
- 验证测试通过

**REFACTOR 阶段**:
- 改进代码质量
- 保持测试通过
- 验证功能不变

**重点强调**:
- ⚠️ **绝对禁止**先写代码再写测试
- ✅ **必须**先写失败测试，理解为什么失败
- ✅ **必须**在 GREEN 阶段后才能重构

### 2.5 多 Skill 协作展示

[系统架构师要求] 展示工作流中 Skill 的自动切换

2. **Spec Skill**: 生成功能规格
3. **Implementer Skill**: 实现代码

## 4. 约束与风险

### 4.1 技术约束

| 约束项 | 影响 | 缓解策略 |
|--------|------|---------|
| 真实项目复杂度 | 分散对工作流的关注 | 渐进式真实度（初级→高级） |
| 平台差异 | 命令在不同平台行为不同 | 标注平台特定行为 |
| 依赖管理 | 示例项目依赖可能过时 | 使用锁定版本，定期验证 |

### 4.2 内容风险

| 风险 | 影响 | 缓解策略 |
|------|------|---------|
| 过度简化 | 用户无法应用到真实项目 | 初级用简化项目，高级用真实项目 |
| 缺乏验证 | 工作流无法复现 | 每步提供验证命令 |
| TDD 合理化 | 用户跳过 RED 阶段 | 强调"先写失败测试"的重要性 |

### 4.3 教学风险

| 风险 | 缓解策略 |
|------|---------|
| 工作流太长 | 分步骤，提供时间估算 |
| 概念缺失 | 每个工作流引用前置概念 |
| 错误处理不足 | 每个工作流包含故障排除章节 |

## 附录 A: TDD 工作流 ASCII 流程

```text
TDD 工作流 - RED-GREEN-REFACTOR

┌────────────────────────────────────────────────────┐
│ RED: 写失败的测试                                    │
└────────────────┬───────────────────────────────────┘
                 │
                 │ 1. 理解需求
                 ▼
         ┌───────────────┐
         │ 编写测试用例   │
         └───────┬───────┘
                 │
                 │ 2. 运行测试（预期失败）
                 ▼
         ┌───────────────┐
         │ 验证测试失败   │ ← ⚠️ 必须确认失败原因
         └───────┬───────┘
                 │ ✅ 测试失败
                 ▼
┌────────────────────────────────────────────────────┐
│ GREEN: 编写最小代码使测试通过                         │
└────────────────┬───────────────────────────────────┘
                 │
                 │ 3. 编写最小实现
                 ▼
         ┌───────────────┐
         │ 运行测试       │
         └───────┬───────┘
                 │
                 │ 4. 迭代直到通过
                 ▼
         ┌───────────────┐
         │ 验证测试通过   │ ← ✅ 测试通过即可，不追求完美
         └───────┬───────┘
                 │ ✅ 测试通过
                 ▼
┌────────────────────────────────────────────────────┐
│ REFACTOR: 改进代码质量                                │
└────────────────┬───────────────────────────────────┘
                 │
                 │ 5. 重构代码
                 ▼
         ┌───────────────┐
         │ 保持测试通过   │ ← ⚠️ 重构时测试必须保持通过
         └───────┬───────┘
                 │ ✅ 代码改进且测试通过
                 ▼
         ┌───────────────┐
         │ 完成功能       │
         └───────────────┘

⚠️ 禁止：先写代码再写测试（伪 TDD）
⚠️ 禁止：跳过 RED 阶段
✅ 必须：先理解失败原因，再编写实现
```
