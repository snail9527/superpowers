# F-006 角色导向学习路径 UX 分析

**功能 ID**: F-006
**功能名称**: 角色导向学习路径
**分析者**: UX 专家
**文档版本**: 1.0
**分析日期**: 2026-03-30

---

## 1. 功能概述

**目标**：为不同开发者角色提供个性化的学习建议，满足特定角色的学习需求。

**核心价值**：
- 针对性内容推荐
- 优化的学习顺序
- 角色特定的最佳实践

---

## 2. 用户角色定义

### 2.1 角色分类体系

**主要角色**：

**一线开发者**（Front-line Developer）
- 职责：日常代码开发、功能实现、Bug 修复
- 关注点：快速上手、常见任务、故障排除
- 时间预算：碎片化学习（每次 10-30 分钟）

**技术负责人**（Tech Lead）
- 职责：架构设计、技术选型、代码审查
- 关注点：系统架构、质量保证、团队协作
- 时间预算：深度学习（每次 30-60 分钟）

**扩展开发者**（Extension Developer）
- 职责：自定义 Skill 开发、系统集成、贡献代码
- 关注点：核心 API、扩展机制、最佳实践
- 时间预算：项目导向学习（按需学习）

**运维/DevOps 工程师**（DevOps Engineer）
- 职责：部署、监控、维护、性能优化
- 关注点：部署流程、监控配置、故障处理
- 时间预算：问题导向学习（遇到问题时）

### 5.1 一线开发者最佳实践

**高效工作流**：
```
日常开发流程

1. 需求确认
   → 使用 brainstorm 理解需求

2. 快速原型
   → 使用 develop 快速实现

3. 质量检查
   → 使用 test 和 review 验证

4. 问题修复
   → 使用 debug 快速定位

时间优化：
- 每个阶段 15-30 分钟
- 优先自动化流程
- 建立个人模板
```

**常见模式**：
```
模式 1：迭代开发
brainstorm → plan → develop → test → review
（适用于新功能开发）

模式 2：快速修复
debug → fix → test → review
（适用于 Bug 修复）

模式 3：优化重构
analyze → refactor → test → review

```

## 8. 设计决策记录

### 8.1 角色数量决策

**决策**：4 个主要角色（一线开发者、技术负责人、扩展开发者、DevOps 工程师）

**理由**：
- 覆盖 SuperPower 的主要用户群体
- 角色间有明确区分
- 不至于过于细分导致维护困难

**风险**：某些用户可能不严格属于任何角色。

**缓解**：
- 允许多角色选择
- 提供混合路径
- 强调角色建议而非强制

### 8.2 角色特定内容决策

**决策**：提供角色特定的学习路径，但共享核心内容

**理由**：
- 避免内容重复
- 保持一致性
- 降低维护成本

**实现**：
- 核心概念：所有角色共享
- 学习顺序：按角色调整
- 实践示例：按角色定制
- 最佳实践：按角色特定
