# 跨角色综合分析 - SuperPower 学习教程

**分析日期**: 2026-03-30
**分析范围**: 4 个角色视角的综合分析
**涉及角色**: UX 专家、系统架构师、UI 设计师、技术领域专家

---

## 执行摘要

### 核心共识
1. **渐进式学习路径**：所有角色认同"快速入门 → 核心概念 → 实战应用 → 高级主题"的渐进式结构
2. **实践导向**：强调代码示例的可执行性和真实性
3. **质量保证**：TDD 和代码审查作为核心流程

### 关键冲突
1. **深度 vs 广度**：技术专家要求深入机制，UX 要求控制认知负荷
2. **文档架构**：架构师强调可扩展性，UI 设计师强调纯文本约束
3. **学习路径设计**：线性 vs 模块化的优先级分歧

### 主要增强建议
- EP-001: 渐进式深度标记机制
- EP-002: 代码示例分层验证
- EP-003: 可视化与纯文本的混合策略
- EP-004: 概念依赖图导航

## 增强建议（Enhancement Proposals）

### EP-001: 渐进式深度标记机制

**理由描述**：
解决技术深度要求与认知负荷管理的冲突。技术领域专家要求深入解释 Agent/Skill/Command 机制（如"零上下文继承"），而 UX 专家强调"每章最多 3 个新概念"的认知负荷约束。

**实施方案**：
- 在每个概念中使用 **📊 深度标记**：
  - `📊 基础层`：必须理解的核心概念（5-10 分钟）
  - `📊 进阶层`：完整技术细节（15-20 分钟）
  - `📊 精通层`：高级模式和边缘案例（按需学习）

- **示例结构**：
  ```markdown
  ## Agent 机制

  ### 📊 基础层：什么是 Agent？
  [简洁解释 + 基本示例]

  ### 📊 进阶层：子代理调度策略
  [技术细节 + 状态机图]

  > 💡 **新用户提示**：可以跳过进阶层，先完成实战示例
  > 💡 **高级用户提示**：深入理解有助于自定义工作流
  ```

**影响功能**: F-002（核心概念讲解）、F-007（高级主题深入）

**相关角色**:
- UX 专家：支持分层披露原则
- 技术领域专家：确保深度内容不被删除
- 系统架构师：需支持内容分层的数据模型

**置信度**: [RESOLVED]
- 权衡：平衡新用户的认知负荷与高级用户的技术深度需求
- 实施成本：中等（需修改文档模板和元数据 schema）
- 风险：低（向后兼容，基础层内容满足所有用户最低需求）

### EP-002: 代码示例分层验证

**理由描述**：
技术领域专家强调"每个概念都配实际可运行的示例"，系统架构师要求"代码示例可执行性验证"，但两者在验证粒度上存在分歧。架构师要求 CI/CD 自动验证，技术专家强调真实项目场景的验证。

**实施方案**：
- **三级验证体系**：
  1. **语法验证**（L1）：Markdown lint + TypeScript 类型检查
     - 工具：`markdownlint-cli` + `tsc --noEmit`
     - 触发：每次提交

  2. **执行验证**（L2）：关键代码片段实际执行
     - 工具：`vitest` 单元测试
     - 覆盖：核心概念的示例代码（标记 `@verified`）
     - 触发：PR 合并前

  3. **场景验证**（L3）：真实项目集成测试
     - 工具：端到端测试 + 手动验证
     - 覆盖：常见工作流实战示例
     - 触发：每个发布周期

- **标记方案**：
  ```markdown
  ```typescript title="src/core/agent.ts" @verified
  // 此代码已通过 L2 验证
  ```

  ```typescript
  // 此代码仅供参考，未标记验证级别
  ```
  ```

**影响功能**: F-001（快速入门教程）、F-003（常见工作流实战）

**相关角色**:
- 系统架构师：主导 CI/CD 集成
- 技术领域专家：定义场景验证标准
- UI 设计师：需在文档中显示验证标记

**置信度**: [RESOLVED]
- 权衡：自动化验证成本 vs 内容质量保证
- 实施成本：高（需搭建测试基础设施）
- 风险：中等（验证失败可能阻塞文档发布）

### EP-003: 可视化与纯文本的混合策略

**理由描述**：
UI 设计师要求"纯文本约束下的视觉设计"（ASCII 图表，76 字符宽度），但 UX 专家强调"可视化图表 + 类比"的必要性，技术领域专家的架构分析依赖图表（状态机、依赖图）。纯文本限制可能损害复杂概念的可理解性。

**实施方案**：
- **分层可视化策略**：
  1. **核心层（ASCII）**：所有图表提供 ASCII 版本
     - 确保离线可用和终端兼容
     - 遵循 76 字符宽度限制

  2. **增强层（Mermaid）**：复杂图表提供 Mermaid 代码
     - 支持静态站点生成器的自动渲染
     - 降级机制：Mermaid 渲染失败时显示 ASCII

  3. **参考层（外部）**：高度复杂的架构图链接到外部工具
     - 使用 Excalidraw、Figma 等工具
     - 明确标记为"可选补充材料"

- **示例结构**：
  ```markdown
  ### 子代理调度流程

  **ASCII 版本**：
  ```text
  ┌─────────┐
  │ Agent   │
  └────┬────┘
       │
       ▼
  ```

  **增强版**（在线查看）：
  ```mermaid
  graph TD
    A[Agent] --> B[Subagent]
  ```

  **交互版**（可选）：[在 Excalidraw 中查看]
  ```

**影响功能**: F-002（核心概念讲解）、所有文档

**相关角色**:
- UI 设计师：制定 ASCII 图表规范
- UX 专家：评估图表可理解性
- 系统架构师：选择静态站点生成器（需支持 Mermaid）

**置信度**: [SUGGESTED]
- 权衡：复杂度 vs 可访问性
- 实施成本：中等（需选型并配置渲染工具）
- 风险：低（ASCII 作为 fallback 保证兼容性）
- 需要用户决策：是否允许依赖外部渲染库

### EP-004: 概念依赖图导航

**理由描述**：
系统架构师提出"知识图谱构建"和"概念依赖关系显式建模"，UX 专家强调"导航清晰性"和"跨章节链接"，但两者在导航实现上存在分歧。架构师强调自动化推荐，UX 专家强调用户主动探索。

**实施方案**：
- **显式依赖标记**：
  - 在 Front matter 中添加 `dependencies` 和 `relatedConcepts` 字段：
    ```yaml
    ---
    id: core-concepts-agents
    title: Agent 机制
    level: beginner
    dependencies: [quick-start]
    relatedConcepts: [skills, commands, task-context]
    estimatedTime: 15
    ---
    ```

- **三类导航组件**：
  1. **前置要求检查**：页面顶部显示未完成的前置概念
     ```
     ⚠️ 前置要求：快速入门教程（未完成）
     [跳转到快速入门 →]
     ```

  2. **相关概念推荐**：侧边栏显示相关概念
     ```
     相关概念
     • Skill 触发机制 →
     • Command 发送流程 →
     ```

  3. **学习路径图**：提供依赖关系的可视化视图
     ```text
     快速入门
         ↓
     核心概念
     ├── Agent 机制 ← 当前
     ├── Skill 机制
     └── Command 系统
     ```

**影响功能**: F-002（核心概念讲解）、F-006（角色导向学习路径）

**相关角色**:
- 系统架构师：实现依赖图数据模型和自动化推荐
- UX 专家：设计导航交互和视觉呈现
- UI 设计师：确保导航组件的视觉一致性

**置信度**: [RESOLVED]
- 权衡：自动化推荐的准确性 vs 用户自主性
- 实施成本：中等（需扩展 Front matter schema 和导航组件）
- 风险：低（显式导航不影响现有线性阅读）

## 功能冲突映射

### F-001: 快速入门教程（10分钟）

#### 共识（Consensus）
- **目标一致**：所有角色认同"首次成功体验"是关键转化点
- **时间约束**：10 分钟内完成安装 → 验证 → 第一个命令
- **简化策略**：最小化步骤，每个都有验证

#### 冲突（Conflicts）

**决策点 1: 技术深度控制**
- **技术领域专家观点**：
  - 必须解释"为什么"（如为什么用子代理而非直接调用）
  - 警告：过度简化可能导致后续理解困难

- **UX 专家观点**：
  - 10 分钟内只展示"怎么做"，不解释"为什么"
  - 原则：最小认知负荷，每步最多 3 个新概念

- **解决方案**: EP-001 渐进式深度标记
  - 快速入门使用 `📊 基础层` 内容
  - 提供"深入理解"链接到进阶层
  - 置信度: [RESOLVED]

**决策点 2: 错误处理策略**
- **系统架构师观点**：
  - 必须提供完整的故障排除清单
  - CI/CD 验证所有命令示例可执行

- **UX 专家观点**：
  - 错误信息应"可操作"而非"技术化"
  - 首次失败时提供简化恢复路径

- **解决方案**: 两层错误处理
  - 第一层：简化恢复（"尝试这个修复"）
  - 第二层：完整故障排除（展开查看详情）
  - 置信度: [RESOLVED]

- **前置**: 无
- **后续**: F-002（核心概念讲解）
- **依赖**: 安装脚本可用性、Node.js 环境
- **相关**: F-005（任务导向指南 - "首次任务"场景）

### F-002: 核心概念讲解（Agent/Skill/Command）

#### 共识（Consensus）
- **核心三要素**：Agent、Skill、Command 是必须讲解的核心
- **类比必要性**：使用类比降低抽象概念理解难度
- **实例驱动**：每个概念配可运行示例

#### 冲突（Conflicts）

**决策点 1: 概念覆盖深度**
- **技术领域专家观点**：
  - Agent：必须深入"零上下文继承"、"状态机"、"双阶段审查"
  - Skill：必须解释"Rigid vs Flexible"、"1% 触发规则"
  - Command：必须展示"Front Matter 解析"、"自动匹配"

- **UX 专家观点**：
  - 遵循"每章 3 概念"规则
  - Agent/Skill/Command 各占一章，每章内部分层
  - 避免单章信息过载

- **UI 设计师观点**：
  - 纯文本约束限制复杂图表展示
  - ASCII 图表 76 字符宽度限制

- **解决方案**: EP-001 + EP-003
  - 每个概念分基础/进阶/精通三层
  - ASCII 图表 + Mermaid 增强版
  - 置信度: [RESOLVED]

**决策点 2: 概念讲解顺序**
- **技术领域专家观点**：
  - 推荐：Agent → Skill → Command（技术实现顺序）
  - 理由：理解 Agent 后才能理解 Skill 如何触发 Agent

- **UX 专家观点**：
  - 推荐：Command → Skill → Agent（用户认知顺序）
  - 理由：用户先"发送命令"，看到"技能触发"，再理解"代理执行"

- **解决方案**: 并行讲解 + 依赖图
  - 提供"技术实现路径"和"用户认知路径"两种顺序
  - 使用 EP-004 概念依赖图让用户自主选择
  - 置信度: [SUGGESTED]
  - **需要用户决策**: 默认顺序选择

### F-003: 常见工作流实战

#### 共识（Consensus）
- **真实性**：使用真实项目场景，避免玩具示例
- **完整性**：展示"问题 → 方案 → 验证"完整流程
- **多样性**：覆盖不同类型的工作流（debug/feature/refactor）

#### 冲突（Conflicts）

**决策点 1: 工作流选择标准**
- **技术领域专家观点**：
  - 必须包含 TDD 工作流（展示 RED-GREEN-REFACTOR）
  - 必须包含调试工作流（展示"先写失败测试"）
  - 理由：这些是 SuperPower 的核心价值

- **系统架构师观点**：
  - 应包含集成工作流（多 Skill 协作）
  - 应包含扩展工作流（自定义 Skill）
  - 理由：展示可扩展性和架构能力

- **UX 专家观点**：
  - 应按"任务频率"排序（最常见的任务优先）
  - 应提供"任务索引"支持快速查找
  - 理由：用户带着具体问题来查找

- **解决方案**: 混合策略
  - P0（必须）：TDD、Debug、Feature（技术专家）
  - P1（应该）：集成、Refactor（架构师）
  - P2（可选）：按任务类型提供索引（UX）
  - 置信度: [RESOLVED]
  - 权衡: 技术深度 vs 用户实用性

**决策点 2: 示例项目选择**
- **技术领域专家观点**：
  - 使用真实开源项目（如 SuperPower 自身）
  - 避免过度简化导致"理想化"示例

- **UX 专家观点**：
  - 使用简化但真实的项目（如 TODO 应用）
  - 理由：真实项目复杂度可能分散对工作流的关注

- **解决方案**: 渐进式真实度
  - 初级工作流：简化项目（专注流程）
  - 高级工作流：真实项目（展示复杂度）
  - 置信度: [RESOLVED]

### F-004: 质量保证流程（TDD + 代码审查）

#### 共识（Consensus）
- **TDD 重要性**：所有角色认同 TDD 是核心流程
- **强制性**：强调"测试先写"的不可协商性
- **双阶段审查**：规范符合性 + 代码质量

#### 冲突（Conflicts）

**决策点 1: TDD 讲解深度**
- **技术领域专家观点**：
  - 必须深入"为什么 TDD"（反驳常见合理化）
  - 必须展示"测试反模式"（伪 TDD、测试后写）
  - 理由：用户有强烈动机跳过 TDD

- **UX 专家观点**：
  - 优先展示"怎么做"，"为什么"作为补充材料
  - 避免过度说教导致用户反感
  - 理由：用户想快速上手，不是被说服

- **解决方案**: 分层说服
  - 快速入门：只展示"怎么做"
  - 核心概念：简要解释"为什么"
  - 深度主题：完整的"常见合理化反驳"
  - 置信度: [RESOLVED]

**决策点 2: 代码审查流程展示**
- **技术领域专家观点**：
  - 必须展示两阶段审查的提示词模板
  - 必须解释"规范符合性"与"代码质量"的区别

- **系统架构师观点**：
  - 应强调审查的状态机（DONE/DONE_WITH_CONCERNS/...）
  - 应展示审查循环的终止条件

- **UX 专家观点**：
  - 对新手用户，两阶段审查可能过于复杂
  - 建议首次体验时简化为单一审查

- **解决方案**: 渐进式复杂度
  - 首次接触：单一审查（合并两个阶段）
  - 核心概念：两阶段审查（完整流程）
  - 高级主题：自定义审查策略
  - 置信度: [SUGGESTED]

- **内部关联**: TDD 流程 → F-003（常见工作流）

### F-005: 任务导向指南

#### 共识（Consensus）
- **可发现性**：支持用户快速找到"如何完成 X"
- **任务分类**：按用户意图分类（添加功能/修复 Bug/重构）
- **独立性**：每个指南可独立学习

#### 冲突（Conflicts）

**决策点 1: 索引组织方式**
- **UX 专家观点**：
  - 按"用户意图"分类（我想做什么？）
  - 示例：添加登录功能、修复性能问题、重构代码

- **技术领域专家观点**：
  - 按"技术模式"分类（我需要什么模式？）
  - 示例：TDD 工作流、调试工作流、集成工作流

- **系统架构师观点**：
  - 按"SuperPower 能力"分类（什么工具能帮我？）
  - 示例：Brainstorm、Implementer、Reviewer 技能

- **解决方案**: 多维索引
  - 主索引：用户意图（UX 优先）
  - 次级索引：技术模式（技术专家）
  - 侧边栏：相关技能（架构师）
  - 置信度: [RESOLVED]

**决策点 2: 指南内容结构**
- **UX 专家观点**：
  - 结构：问题诊断 → 解决方案 → 验证
  - 强调"如何判断这是正确的方案"

- **技术领域专家观点**：
  - 结构：前置检查 → 执行步骤 → 常见错误
  - 强调"如何避免典型错误"

- **解决方案**: 混合模板
  - 顶部：问题诊断（UX）
  - 中部：执行步骤 + 前置检查（技术专家）
  - 底部：验证 + 常见错误（两者）
  - 置信度: [RESOLVED]

### F-006: 角色导向学习路径

#### 共识（Consensus）
- **个性化价值**：不同角色有不同学习需求
- **角色定义**：需要明确目标用户角色（新手/中级/高级）

#### 冲突（Conflicts）

**决策点 1: 角色分类方式**
- **UX 专家观点**：
  - 按"经验水平"分类（新手/中级/高级）
  - 理由：学习路径主要依赖经验

- **技术领域专家观点**：
  - 按"技术背景"分类（前端/后端/全栈）
  - 理由：不同背景有不同技术盲点

- **系统架构师观点**：
  - 按"使用场景"分类（个人开发/团队协作/企业级）
  - 理由：场景影响 SuperPower 的使用方式

- **解决方案**: 矩阵式路径
  - 主维度：经验水平（新手/中级/高级）
  - 次维度：技术背景（可选标签）
  - 场景标签：个人/团队/企业（标记在指南上）
  - 置信度: [SUGGESTED]
  - **需要用户决策**: 是否支持多维度分类（增加复杂度）

**决策点 2: 路径内容编排**
- **UX 专家观点**：
  - 每个角色路径应该是"精选"而非"完整"
  - 避免 overwhelm，只推荐最相关的内容

- **技术领域专家观点**：
  - 每个角色路径应该是"完整"的
  - 确保不会因为"精选"而遗漏关键概念

- **解决方案**: 核心 + 可选
  - 核心路径：必须完成的内容（80% 用户需要）
  - 可选扩展：按需添加的内容（20% 用户需要）
  - 置信度: [RESOLVED]

#### 交叉引用（Cross-References）
- **依赖**: F-001 到 F-005（所有前置功能）
- **内部关联**: 路径引用多个功能和指南
- **相关**: F-007（高级主题 - 针对高级角色）

### F-007: 高级主题深入

#### 共识（Consensus）
- **按需学习**：高级主题不应该是强制路径
- **深度覆盖**：提供完整的技术细节和边缘案例
- **参考性质**：作为"深入理解"的查阅资料

#### 冲突（Conflicts）

**决策点 1: 内容覆盖范围**
- **技术领域专家观点**：
  - 必须包含：子代理提示词工程、自定义 Skill、性能优化
  - 理由：这些是真正"高级"用户需要的内容

- **系统架构师观点**：
  - 必须包含：扩展开发、插件架构、集成模式
  - 理由：展示 SuperPower 的可扩展性

- **UX 专家观点**：
  - 必须包含：调试高级问题、故障排除、性能调优
  - 理由：高级用户遇到的常见实际问题

- **解决方案**: 主题分类
  - 技术深度：提示词工程、自定义 Skill（技术专家）
  - 架构扩展：插件开发、集成模式（架构师）
  - 实战技巧：调试、调优、故障排除（UX）
  - 置信度: [RESOLVED]

**决策点 2: 讲解方式**
- **技术领域专家观点**：
  - 深入源码，解释实现细节
  - 展示"为什么这样设计"

- **UX 专家观点**：
  - 保持"问题导向"的讲解风格
  - 即使高级主题，也应该解决实际问题

- **解决方案**: 问题驱动 + 源码深入
  - 顶部：实际问题场景
  - 中部：解决方案（用户视角）
  - 底部：源码实现（深入理解，可选）
  - 置信度: [RESOLVED]

- **内部关联**: 高级主题引用核心概念的进阶层
