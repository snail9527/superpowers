# SuperPower 学习教程 - 指导规格说明

**文档版本**: 1.0
**创建日期**: 2026-03-30
**目标受众**: 中级用户（有 AI 编程经验）

---

## 1. 项目定位与目标

### 1.1 项目概述

为 SuperPower 开源项目创建详尽的学习教程，帮助中级开发者深入理解 SuperPower 的核心思想、实施流程、重要概念（Agent、Skill、Command）以及文档产物和归档整理结构。

### 1.2 目标受众

CONFIRMED: 中级用户（有 AI 编程经验）
- 已了解基本的 AI 编程助手概念
- 有使用 Claude Code、Cursor 或类似工具的经验
- 希望系统化学习 SuperPower 的工作流和最佳实践

### 1.3 成功标准

- 新用户在 10 分钟内完成首次成功操作
- 30 分钟内理解三大核心概念（Agent、Skill、Command）
- 1 小时内能完成简单工作流
- 每个教程都可独立运行和验证
- 所有示例命令都经过测试验证

## 2. 概念与术语

| 术语 (英文) | 术语 (中文) | 定义 | 类别 |
|-----------|-----------|------|------|
| Agent | 代理/智能体 | AI 编程助手，能够理解指令、执行任务、使用工具的 AI 实体 | 核心 |
| Skill | 技能 | 可组合的模块化工作单元，定义了 Agent 如何处理特定类型任务 | 核心 |
| Command | 命令 | 用户向 Agent 发出的指令请求，触发相应的 Skill 执行 | 核心 |
| Subagent | 子代理 | 为执行特定独立任务而分派的新 Agent 实例，具有隔离的上下文 | 技术 |
| Workflow | 工作流 | 由多个 Skill 组成的端到端流程，定义了完整的任务执行序列 | 核心 |
| TDD | 测试驱动开发 | 先编写失败的测试，再编写最小代码使测试通过的开发方法论 | 技术 |
| Spec Compliance | 规范遵循 | 确保实现严格符合设计规范的质量检查流程 | 技术 |
| MCP | 模型上下文协议 | 标准化的 AI 模型与工具/数据源之间的通信协议 | 技术 |

## 4. UX 专家决策

### 4.1 学习路径设计

CONFIRMED: **渐进式披露路径**

教程应按照以下顺序组织：

1. **快速入门**（10 分钟）
   - 安装与验证
   - 第一个命令
   - 验证安装成功

2. **核心概念**（30 分钟）
   - Agent 系统架构
   - Skill 机制
   - Command 结构

3. **常见工作流**（1 小时）
   - 工作流编排基础
   - 多模型协作
   - 调试和故障排除

4. **高级主题**（按需学习）
   - Agent 机制深度
   - Skill 设计模式
   - 工作流质量保证

### 4.2 内容组织原则

MUST:
- 每个章节独立完整，可单独学习
- 使用清晰的进度指示
- 提供前后章节的导航链接

SHOULD:
- 使用实际命令示例
- 提供可复制的代码块
- 包含预期输出展示

MAY:
- 添加挑战练习
- 提供扩展阅读链接

## 6. UI 设计师决策

### 6.1 视觉元素规范

CONFIRMED: **ASCII 图表**

所有流程图、架构图、关系图 MUST 使用 ASCII 艺术格式。

示例：
```
User (Command)
    ↓
Agent (主代理)
    ↓
Skill (触发条件匹配)
    ↓
Workflow (执行流程)
```

### 6.2 代码展示规范

MUST:
- 使用语法高亮的代码块
- 标注代码语言（bash, markdown, javascript 等）
- 命令示例使用 `$` 前缀
- 输出示例使用不带前缀的代码块

示例：
```bash
$ superpower brainstorm "Add user authentication"
```

SHOULD:
- 长命令添加续行符 `\`
- 关键参数加粗说明
- 包含预期输出

### 6.3 文档格式规范

MUST:
- 使用 H1 (`#`) 作为文档标题
- 使用 H2 (`##`) 作为主要章节
- 使用 H3 (`###`) 作为子章节
- 列表使用 `-` 或 `1.` 格式

SHOULD:
- 每个章节不超过 500 字
- 使用表格展示对比信息
- 重要概念使用**加粗**标注

## 9. 风险与约束

### 9.1 技术约束

1. **平台差异**: SuperPower 在不同平台（Claude Code、Cursor、Gemini CLI）上有行为差异
   - 缓解策略：标注平台特定行为，提供通用解决方案

2. **版本迭代**: SuperPower 持续更新，教程可能过时
   - 缓解策略：记录版本号，提供更新检查机制

### 9.2 内容风险

1. **信息过载**: 一次性解释太多概念
   - 缓解策略：严格遵循渐进式披露，每章最多 3 个新概念

2. **缺乏实践**: 只有理论没有操作
   - 缓解策略：每个概念后都跟可执行的示例

3. **导航混乱**: 模块化后用户迷失
   - 缓解策略：提供清晰的学习路径图和角色导向指南

## 附录 A: RFC 2119 关键词说明

本文档使用 RFC 2119 定义的以下关键词来表示需求等级：

- **MUST**: 绝对必须，无例外
- **MUST NOT**: 绝对禁止
- **SHOULD**: 强烈推荐，除非有充分理由不这样做
- **SHOULD NOT**: 强烈不推荐，除非有充分理由这样做
- **MAY**: 可选
