# 跨功能技术决策分析

## Agent 机制深度剖析

### 架构模式：有限状态子代理系统

SuperPower 的子代理不是简单的任务分发，而是一个完整的**分布式质量保证系统**。

#### 核心设计原则

**1. 零上下文继承（Zero Context Inheritance）**
```
主会话 (Coordinator)
    ↓ 精心构建的上下文
子代理 (Implementer) [干净状态]
    ↓ 完成任务
子代理 (Spec Reviewer) [干净状态]
    ↓ 审查规范符合性
子代理 (Code Reviewer) [干净状态]
    ↓ 审查代码质量
返回主会话
```

**技术价值：**
- 避免上下文污染导致的决策不一致
- 每个子代理基于相同的信息做决策
- 主会话保持轻量，专注于协调

**2. 双阶段审查（Two-Stage Review）**

**阶段 1：规范符合性审查**
```markdown
审查清单：
- [ ] 所有要求都已实现？
- [ ] 没有添加未要求的功能？
- [ ] 验证步骤是否可执行？
```

**阶段 2：代码质量审查**
```markdown
审查清单：
- [ ] 代码可读性
- [ ] 测试覆盖率
- [ ] 错误处理
- [ ] 性能考虑
```

**技术价值：**
- 分离关注点：先保证"做对的事"，再保证"做的事对"
- 防止过早优化：规范审查通过前不考虑质量
- 审查循环：发现问题 → 同一子代理修复 → 重新审查

**3. 渐进式模型选择（Progressive Model Selection）**

```
任务复杂度 → 模型选择
├─ 机械实现 (1-2 文件) → 快速廉价模型
├─ 集成判断 (多文件)   → 标准模型
└─ 架构设计 (全局)     → 最强模型
```

**技术价值：**
- 成本优化：简单任务不用昂贵模型
- 速度优化：机械任务用快速模型
- 质量保证：复杂任务用强力模型

#### 子协议状态机

```
                    ┌─────────────────┐
                    │  NEEDS_CONTEXT  │
                    │  (需要更多上下文)  │
                    └────────┬────────┘
                             │ 提供信息
                             ↓
┌─────────────────┐    ┌─────────────────┐
│  DONE (完成)     │ ←─ │    DISPATCH     │
└────────┬────────┘    │  (分派子代理)     │
         │             └────────┬────────┘
         │ 审查                   │
         ↓                      ↓
┌─────────────────┐    ┌─────────────────┐
│  DONE_WITH_     │    │   BLOCKED       │
│  CONCERNS       │    │  (被阻塞)        │
│  (有疑虑完成)     │    └─────────────────┘
└─────────────────┘
```

**状态处理：**
- **DONE**: 直接进入审查流程
- **DONE_WITH_CONCERNS**: 判断疑虑性质，决定是否需要处理
- **NEEDS_CONTEXT**: 提供缺失信息，重新分派同一模型
- **BLOCKED**: 评估阻塞原因，可能需要升级给人或拆分任务

### 与传统模式的对比

| 维度 | 传统 AI 编程 | SuperPower 子代理 |
|------|-------------|------------------|
| 上下文 | 继承主会话 | 零继承 |
| 审查 | 可选/最后 | 强制/每个任务 |
| 模型选择 | 统一模型 | 渐进式 |
| 失败处理 | 重试/忽略 | 状态机驱动 |
| 质量保证 | 事后 | 实时 |

## Skill 设计模式总结

### 技能分类体系

**1. 流程技能（Process Skills）**
- brainstorming（头脑风暴）
- systematic-debugging（系统调试）
- writing-plans（计划编写）

特点：决定如何接近任务，优先级最高

**2. 实现技能（Implementation Skills）**
- test-driven-development（测试驱动开发）
- subagent-driven-development（子代理驱动开发）
- executing-plans（计划执行）

特点：指导具体执行

**3. 协作技能（Collaboration Skills）**
- requesting-code-review（请求代码审查）
- receiving-code-review（接收代码审查）
- using-git-worktrees（使用 Git 工作树）

特点：支持团队协作

### 技能触发机制

**Front Matter 驱动：**
```yaml
---
name: test-driven-development
description: Use when implementing any feature or bugfix,
             before writing implementation code
---
```

**触发算法：**
```
用户输入 → 技能匹配度评估 → ≥ 1% ? → 触发技能
                              ↓ 否
                          继续执行
```

**红旗检测（Rationalization Prevention）：**
```
"这只是简单问题" → 问题是任务 → 检查技能
"我需要更多上下文" → 技能检查先于明确问题
"我记得这个技能" → 技能会演进 → 读取当前版本
```

### Rigid vs Flexible 技能

**Rigid 技能（严格遵循）：**
- TDD：RED-GREEN-REFACTOR 顺序不可改变
- 调试：四阶段根因分析必须完成

**Flexible 技能（灵活适配）：**
- 设计模式：根据具体情境调整
- 代码风格：遵循项目现有规范

## 质量保证流程体系

### 三层质量门控

```
┌──────────────────────────────────────┐
│  第一层：TDD 强制门控                  │
│  - 失败测试必须在生产代码之前           │
│  - Watch it fail 是强制步骤            │
│  - 测试前写代码必须删除                 │
└──────────────────────────────────────┘
              ↓
┌──────────────────────────────────────┐
│  第二层：双阶段审查门控                 │
│  - 规范符合性：只做要求的事              │
│  - 代码质量：正确地做事                 │
│  - 审查循环：问题必须修复并重新审查        │
└──────────────────────────────────────┘
              ↓
┌──────────────────────────────────────┐
│  第三层：最终代码审查                   │
│  - 全局一致性检查                       │
│  - 集成问题发现                         │
│  - 文档完整性验证                       │
└──────────────────────────────────────┘
```

### 质量度量指标

**覆盖维度：**
- 功能覆盖：每个功能都有测试
- 场景覆盖：正常、边界、错误场景
- 路径覆盖：所有代码路径都被测试

**质量维度：**
- 可读性：代码和测试都清晰易懂
- 可维护性：易于修改和扩展
- 可测试性：设计支持测试

## 跨功能技术依赖

### 依赖关系图

```
brainstorming (头脑风暴)
    ↓ 产生设计文档
writing-plans (计划编写)
    ↓ 产生实施计划
using-git-worktrees (Git 工作树)
    ↓ 创建隔离环境
subagent-driven-development (子代理驱动开发)
    ├── test-driven-development (TDD)
    ├── requesting-code-review (代码审查)
    └── systematic-debugging (系统调试)
finishing-a-development-branch (完成开发分支)
```

### 关键依赖约束

**1. 设计 → 计划依赖**
- 计划必须基于已批准的设计
- 设计变更需要重新生成计划

**2. 计划 → 执行依赖**
- 计划中的任务必须是自包含的
- 任务之间的依赖必须明确

**3. 执行 → 质量依赖**
- 每个任务必须通过双阶段审查
- 审查未通过不能进入下一个任务

### 技术风险点

**1. 技能触发遗漏**
- 风险：用户或 AI 跳过必要技能
- 缓解：1% 规则 + 红旗检测

**2. 子代理上下文不足**
- 风险：子代理缺少必要信息导致失败
- 缓解：NEEDS_CONTEXT 状态 + 重新分派

**3. 审查循环无限**
- 风险：反复审查无法通过
- 缓解：问题明确化 + 升级机制

**4. TDD 强制执行的阻力**
- 风险：用户寻找绕过方法
- 缓解：详细反驳常见合理化 + 系统性检测

## 技术实现的关键模式

### 1. 状态机驱动
- 子代理状态：DONE/DONE_WITH_CONCERNS/NEEDS_CONTEXT/BLOCKED
- 审查状态：通过/问题/重新审查

### 2. 模板化提示词
- implementer-prompt.md
- spec-reviewer-prompt.md
- code-quality-reviewer-prompt.md

### 3. 渐进式复杂度
- 从单个技能开始
- 逐步组合多个技能
- 最终形成完整工作流

### 4. 反馈循环
- 每个阶段都有验证步骤
- 问题立即被发现和修复
- 质量持续改进

## 技术视角的最佳实践

1. **遵循技能优先级**：流程技能优先于实现技能
2. **尊重强制规则**：TDD 铁律不可违反
3. **理解状态机**：正确处理子代理的各种状态
4. **利用模板**：使用标准提示词模板保证一致性
5. **渐进式学习**：从简单技能开始，逐步掌握复杂工作流
