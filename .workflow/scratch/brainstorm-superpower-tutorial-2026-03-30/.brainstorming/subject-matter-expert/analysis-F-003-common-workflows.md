# F-003: 常见工作流实战技术分析

## 技术深度评估：⭐⭐⭐⭐

**实战定位：** F-003 是将抽象概念转化为具体操作的关键环节。

## 工作流分类体系

### 1. 创造性工作流（Creative Workflows）

**头脑风暴 → 设计 → 计划 → 实现**

**技术要点：**
- **发散-收敛循环**：从模糊想法到精确规格
- **文档驱动**：每个阶段都有可追溯的文档产出
- **质量门控**：每个阶段结束前必须验证

**技术深度：**
```
头脑风暴 (brainstorming)
    ↓ 产出：设计文档
    ↓ 验证：用户确认
计划编写 (writing-plans)
    ↓ 产出：实施计划
    ↓ 验证：无占位符检查
工作树创建 (using-git-worktrees)
    ↓ 产出：隔离环境
    ↓ 验证：测试基线
计划执行 (subagent-driven-development)
    ↓ 产出：功能代码
    ↓ 验证：双阶段审查
分支完成 (finishing-a-development-branch)
    ↓ 产出：可合并代码
    ↓ 验证：完整测试
```

### 2. 修复性工作流（Corrective Workflows）

**问题诊断 → 根因分析 → 修复验证**

**技术要点：**
- **四阶段调试**：观察 → 分析 → 验证 → 修复
- **测试先行**：bug 修复必须从失败测试开始
- **防御深度**：不只修复症状，要找到根因

**技术深度：**
```
系统调试 (systematic-debugging)
    ├─ 阶段 1：观察问题
    │  └─ 产出：问题陈述
    ├─ 阶段 2：根因分析
    │  └─ 工具：root-cause-tracing
    ├─ 阶段 3：验证假设
    │  └─ 工具：condition-based-waiting
    └─ 阶段 4：修复验证
       └─ 工具：defense-in-depth
```

### 3. 协作性工作流（Collaborative Workflows）

**代码审查 → 反馈响应 → 迭代改进**

**技术要点：**
- **结构化反馈**：按严重程度分类问题
- **响应协议**：系统化处理审查意见
- **质量闭环**：每个问题都要解决和验证

**技术深度：**
```
请求代码审查 (requesting-code-review)
    ↓ 提交：实现摘要 + Git SHA
    ↓ 审查：规范符合性 + 代码质量
    ↓ 分类：Critical/Important/Minor/观察
接收代码审查 (receiving-code-review)
    ↓ 分析：审查意见分类
    ↓ 响应：修复 / 解释 / 争议
    ↓ 验证：问题解决确认
```

## 关键工作流详解

### 工作流 1：从想法到实现

**技术复杂度：⭐⭐⭐⭐⭐**

**完整流程：**
```markdown
1. brainstorming（头脑风暴）
   - 苏格拉底式问题明确需求
   - 探索替代方案
   - 分段展示设计供验证
   - 产出：设计文档

2. writing-plans（计划编写）
   - 分解为 2-5 分钟任务
   - 每个任务包含完整信息
   - Self-review 检查占位符
   - 产出：实施计划

3. using-git-worktrees（Git 工作树）
   - 创建隔离分支
   - 运行项目设置
   - 验证测试基线
   - 产出：干净环境

4. subagent-driven-development（子代理驱动开发）
   - 调度子代理执行任务
   - 双阶段审查
   - 迭代修复
   - 产出：实现代码

5. finishing-a-development-branch（完成分支）
   - 验证所有测试
   - 提供合并选项
   - 清理工作树
   - 产出：可合并代码
```

**技术难点：**
1. **任务粒度控制**：太大难以完成，太小效率低下
2. **上下文构建**：子代理需要恰好足够的信息
3. **质量门控**：每个阶段的验证不能绕过

**示例场景：**
```
用户想法："我想添加一个用户头像上传功能"

执行流程：
[brainstorming] "几个问题："
- 支持哪些图片格式？
- 文件大小限制？
- 需要图片压缩吗？
- 存储在哪里？
- ...

[writing-plans]
任务 1：创建上传 API 端点
任务 2：实现图片验证
任务 3：添加图片压缩
任务 4：集成存储服务
...

[subagent-driven-development]
子代理 1：实现任务 1
  → 规范审查：通过
  → 质量审查：发现 3 个问题
  → 修复后重新审查：通过

子代理 2：实现任务 2
  → 规范审查：缺少 PNG 支持
  → 修复后重新审查：通过
  ...
```

### 工作流 2：系统化调试

**技术复杂度：⭐⭐⭐⭐**

**完整流程：**
```markdown
1. 观察阶段
   - 精确描述问题
   - 记录复现步骤
   - 收集相关日志

2. 分析阶段
   - 使用 root-cause-tracing
   - 追踪执行路径
   - 识别异常点

3. 验证阶段
   - 形成假设
   - 使用 condition-based-waiting
   - 确认根因

4. 修复阶段
   - 写失败测试
   - 修复代码
   - 使用 defense-in-depth 验证
```

**技术难点：**
1. **症状 vs 根因**：容易修复症状而忽略根因
2. **假设验证**：需要设计实验验证假设
3. **防御深度**：不只修复当前问题，还要防止类似问题

**示例场景：**
```
问题报告："用户登录后偶尔被踢出"

[观察阶段]
- 问题：session 过期早于预期
- 频率：约 10% 用户
- 环境：生产环境

[分析阶段]
- root-cause-tracing：
  1. 检查 session 配置 → 正常
  2. 检查 token 刷新逻辑 → 发现竞态条件
  3. 检查并发请求处理 → 发现 session 覆盖问题

[验证阶段]
- 假设：并发请求导致 session 覆盖
- 实验：模拟并发登录
- 确认：100% 复现问题

[修复阶段]
- 测试：写并发登录测试
- 修复：添加 session 锁
- 验证：测试通过，生产监控正常
```

### 工作流 3：代码审查循环

**技术复杂度：⭐⭐⭐**

**完整流程：**
```markdown
1. 提交审查请求
   - 实现摘要
   - 相关 Git SHA
   - 自评估问题

2. 规范符合性审查
   - 检查所有要求实现
   - 检查无额外功能
   - 检查验证步骤

3. 代码质量审查
   - 代码可读性
   - 测试覆盖率
   - 错误处理
   - 性能考虑

4. 响应审查意见
   - Critical：必须修复
   - Important：建议修复
   - Minor：可选修复
   - 观察：记录但不处理
```

**技术难点：**
1. **问题分类**：判断问题严重程度
2. **响应策略**：修复 vs 解释 vs 争议
3. **审查循环**：确保问题真正解决

**示例场景：**
```
代码提交：实现了用户头像上传功能

[规范符合性审查]
❌ 问题：
- 缺少：PNG 格式支持（规格要求）
- 额外：添加了裁剪功能（未要求）

[修复]
- 移除裁剪功能
- 添加 PNG 支持

[重新审查]
✅ 规范符合性通过

[代码质量审查]
⚠️ 问题（Important）：
- 文件上传没有大小限制
- 错误处理不完整
- 缺少文件类型验证

[修复]
- 添加 5MB 限制
- 完善错误处理
- 添加类型验证

[重新审查]
✅ 代码质量通过
```

### 2. 调试工作流示例

**目标：** 展示系统化调试的四阶段

```markdown
# 示例：修复内存泄漏

## 阶段 1：观察
- 问题：服务器内存持续增长
- 频率：运行 24 小时后 OOM
- 数据：内存分析报告

## 阶段 2：分析
- root-cause-tracing：
  - 检查事件监听器 → 发现未移除
  - 检查缓存 → 发现无限增长
  - 检查连接池 → 发现连接未释放

## 阶段 3：验证
- 假设：连接未释放导致泄漏
- 实验：添加连接监控
- 确认：连接数持续增长

## 阶段 4：修复
- 测试：写连接泄漏测试
- 修复：添加连接超时和释放
- 验证：测试通过，内存稳定
```

## 潜在陷阱警告

### 1. 工作流选择错误

**陷阱：** 用创造性工作流解决简单问题
- **示例：** 明确的小 bug 修复也走完整 brainstorming
- **后果：** 效率低下，过度工程
- **正确做法：** 直接进入 systematic-debugging

### 2. 阶段跳跃

**陷阱：** 跳过必要的中间阶段
- **示例：** brainstorming 后直接写代码，跳过 planning
- **后果：** 实现不完整，遗漏边界情况
- **正确做法：** 严格按阶段执行，每个阶段验证后再进入下一个

### 3. 质量门控绕过

**陷阱：** "差不多就行"，跳过审查
- **示例：** 代码审查发现问题但不修复
- **后果：** 技术债累积，质量下降
- **正确做法：** 所有质量门控必须通过

### 4. 子代理滥用

**陷阱：** 简单任务也用子代理
- **示例：** 单行修改也调度子代理
- **后果：** 效率低下，开销过大
- **正确做法：** 机械任务直接执行，复杂任务才用子代理

## 工作流选择决策树

```
任务类型？
├─ 新功能开发 → 创造性工作流
├─ Bug 修复 → 修复性工作流
├─ 代码审查 → 协作性工作流
└─ 性能优化 → 分析性工作流

复杂度？
├─ 简单（< 30 分钟）→ 直接执行
├─ 中等（30 分 - 2 小时）→ 单工作流
└─ 复杂（> 2 小时）→ 组合工作流
```

## 学习检查点

学完 F-003 后，学习者应该能够：

1. **选择工作流**
   - 根据任务类型选择合适的工作流
   - 判断是否需要组合多个工作流
   - 评估任务复杂度

2. **执行工作流**
   - 按正确顺序执行各个阶段
   - 通过每个阶段的质量门控
   - 处理阶段间的转换

3. **处理异常**
   - 工作流中断时如何恢复
   - 质量门控失败时如何修复
   - 子代理失败时如何处理

4. **优化效率**
   - 识别可跳过的冗余步骤
   - 合理使用子代理
   - 平衡质量和速度

## 技术深度总结

F-003 是理论到实践的桥梁。重点是：

1. **流程重于速度**：先做对，再做快
2. **质量不可妥协**：每个门控都必须通过
3. **灵活性**：根据任务调整工作流
4. **系统性**：问题复现时走相同流程

通过实际案例和完整示例，让学习者理解 SuperPower 工作流的真正价值。
